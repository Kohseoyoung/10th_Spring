- RESTful API란?
    - REST(Representational State Transfer) 아키텍처 스타일을 따르는 애플리케이션 프로그래밍 인터페이스
        - URI: 인터넷에서 자원을 식별하기 위한 주소
        - REST: 자원을 URI로 표현하고 HTTP 메서드로 처리하는 설계 원칙
        - API: 프로그램 간 기능이나 데이터를 주고받기 위한 인터페이스

    ---

  ### RESTful API 구성요소

    1. **자원(Resource)**
        - REST에서는 모든 데이터를 자원으로 취급하며, URI로 식별
    2. **HTTP 메서드**
        - GET 조회 (예: 사용자 목록 가져오기)
        - POST 생성 (예: 새 사용자 추가)
        - PUT/PATCH 수정 (예: 사용자 정보 업데이트)
        - DELETE 삭제 (예: 사용자 삭제)

  | GET /users | 사용자 목록 조회 |
      | --- | --- |
  | GET /users/1 | 1번 사용자 조회 |
  | POST /users | 사용자 생성 |
  | PATCH /users/1 | 사용자 수정 |
  | DELETE /users/1 | 사용자 삭제 |
    1. **상태 코드 (status code)**
        - 요청의 결과를 나타냄
            - 200 OK: 성공
            - 201 Created: 자원 생성 성공
            - 400 Bad Request: 잘못된 요청
            - 404 Not Found: 자원을 찾을 수 없음
            - 500 Internal Server Error: 서버 오류
    2. **헤더 (Header)**
        - 요청/응답에 대한 메타데이터를 포함
    3. **페이로드(Payload)**
        - 요청 또는 응답에 포함된 실제 데이터

  ### RESTful API의 핵심 원칙

    1. **클라이언트-서버 구조 (Client-Server)**
        - 클라이언트와 서버가 명확히 분리. 이렇게 분리하면 확정성이 높아지며, 관리가 쉬워짐
    2. **무상태성  (Stateless)**
        - 각 요청은 독립적이어야 하며, 서버는 이전 요청의 상태를 기억하지 않음
        - 클라이언트가 필요한 모든 정보를 요청에 포함해야 함
    3. **캐시 가능 (Cacheable)**
        - 응답 데이터에 캐시 가능 여부를 명시하여 클라이언트가 데이터를 재사용할 수 있게 함
    4. **계층화 시스템 (Layered System)**
        - 클라이언트는 서버와 직접 통신하는지, 중간에 프록시나 게이트웨이가 있는지 알 필요가 없음.
        - 프록시:  중간에서 대신 요청해주는 서버
        - 게이트웨이: 서로 다른 시스템이나 네트워크를 이어주는 관문
    5. **일관된 인터페이스 (Uniform Interface)**
        - 자원을 식별하고 조작하는 방식이 일관적
        - 이를 위해 URI로 자원을 식별하고, HTTP 메서드를 사용
        - 일관된 인터페이스 규칙

          (1) 리소스는 URI로 표현

          (2) HTTP 메서드를 활용 (GET, POST, PUT. DELETE)

          (3) 응답 데이터는 기본데이터 포맷으로 JSON 형식을 사용

            <aside>

          {
          "id": 123,
          "name": "홍길동",
          "age": 18
          }

            </aside>

    6. **코드 온 디맨드 (Code on Demand, 선택적)**
        - 서버가 클라이언트에 실행 가능한 코드를 보낼 수 있음
        - 필수 x, 잘 사용 x

  ### RESTful API 장단점

    1. 장점
        - 확장성: 무상태성과 계층화 덕부에 서버를 쉽게 확장할 수 있음
        - 단순성: HTTP를 사용하므로 학습 곡선이 낮고 기존 웹 인프라와 호환
        - 유연성: 다양한 데이터 형식(JSON, XML 등)을 지원
    2. 단점
        - 복잡한 쿼리 처리 어려움: 단순 CRUD(Create, Read, Update, Delete)외 복잡한 작업은 구현이 까다로울 수 있음
        - 오버페칭/언더페칭: 필요 이상의 데이터가 오거나(오버페칭), 필요한 데이터가 부족할 수 있음(언더페칭)
- 멱등성이란?
    - 같은 요청을 여러 번 보내도 결과가 한 번 보낸 것과 동일한 성질
    - **GET:** 여러 번 조회해도 데이터가 바뀌지 않으므로 멱등
    - **PUT:** 같은 내용으로 여러 번 수정해도 결과가 같으므로 멱등
    - **DELETE**: 한 번 삭제하든 여러 번 삭제 요청하든 최종 상태는 삭제이므로 보통 멱등
    - **POST**: 같은 요청을 여러 번 보내면 데이터가 여러 개 생성될 수 있어서 보통 멱등이 아님.
    - **PATCH:** 보통 멱등적으로 설계할 수 있지만, 항상 그런 것은 아님.
        - “name을 민지로 바꿔라” → 여러번 보내도 결과 같음 → 멱등
        - “조회수를 1 증가시켜라” → 보낼 때마다 값이 바뀜 → 멱등 아님
- HTTP 메서드 종류
    - HTTP 메서드는 서버 자원에 대해 조회, 생성, 수정, 삭제 같은 동작을 나타내는 방식
    1. **GET(조회)**
        - GET /users → 사용자 목록 조회
        - GET /users/3 → 3번 사용자 조회
    2. **POST(생성)**
        - POST /usesrs → 새 사용자 생성
    3. **PUT(전체 수정)**
        - PUT /users/3 → 3번 사용자 정보를 전체 교체 수준으로 설정
    4. **PATCH(부분 수정)**
        - PATCH /users/3 → 3번 사용자 정보 일부만 수정
    5. **DELETE(삭제)**
        - DELETE /users/3 → 3번 사용자 삭제

  +추가로 자주 보이는 것

    - HEAD: 헤더만 조회
    - OPTIONS: 지원하는 메서드 확인
